C.0 Cuenta Hetzner Crítico
Decisiones
2FA obligatorio en cuenta
DPA firmado antes de deploy
Registro con datos reales, sin VPN
Qué protege
Acceso a consola web (bypasea SSH y firewalls)
Base legal GDPR Art. 28
Sin esta capa
Phishing en cuenta → acceso total sin SSH
Sin DPA → procesamiento PII sin base legal
↓
C.1 Firewall externo Hetzner — Primera línea Crítico
Reglas
Puerto 22 → solo tu IP fija
Puerto 80/443 → abierto
Todo lo demás → DENY
Qué protege
El tráfico no llega ni al servidor
Elimina SSH brute force antes de UFW
Sin esta capa
Cualquier IP puede intentar SSH
Fail2Ban debe trabajar solo
↓
C.2 Sistema Operativo — Segunda línea Alto impacto
Controles
Usuario no-root
SSH key-only / root deshabilitado
UFW alineado con C.1
Fail2Ban + unattended-upgrades
Detección
AIDE — integridad de binarios
auditd — comportamiento Docker
logwatch — reporte diario
Sin esta capa
Escalada de privilegios desde cualquier acceso
Rootkits invisibles sin AIDE
↓
C.3 Fix Docker/UFW — El gap que nadie menciona Crítico
El problema
Docker ignora UFW via iptables directo
5432:5432 abre PostgreSQL al mundo aunque UFW diga NO
Solución
Eliminar bloque ports en PostgreSQL
Solo N8N tiene ports con bind a 127.0.0.1
Límite de logs Docker: max-size 10m
Sin esta capa
PostgreSQL expuesto a internet entero
UFW activo pero inefectivo para Docker
↓
C.4 Egress Filtering — La defensa contra exfiltración Crítico
Implementación
Cadena DOCKER-USER de iptables
DROP por defecto + ACCEPT 443, 80, 53
ufw deny outgoing NO es suficiente
Qué bloquea
Exfiltración de datos a servidor externo
Reverse shells (puertos arbitrarios)
Descarga de payloads adicionales
Sin esta capa
Atacante dentro → saca datos silenciosamente
Breach GDPR total sin señales
↓
C.5 Silos Docker — Aislamiento por proyecto Alto impacto
Arquitectura
Red interna por proyecto (pa1_internal)
proxy_net solo entre Nginx y N8N
PostgreSQL sin acceso a proxy_net
Hardening contenedor
no-new-privileges:true
user: "1000:1000"
read_only:true + tmpfs /tmp
Límites CPU/RAM por contenedor
Límite conocido
Aislamiento Docker ≠ VM isolation
Escape de contenedor llega al host
Mitigado por C.2 + no-new-privileges
↓
C.6 PostgreSQL — Privilegios mínimos por proyecto Alto impacto
Modelo
1 usuario + 1 DB por proyecto
Nunca usuario postgres por defecto
N8N necesita DDL (TypeORM)
Bloqueos
REVOKE CONNECT entre DBs de proyectos
NOSUPERUSER NOCREATEDB NOCREATEROLE
Verificar: usesuper = false
Resultado
N8N GeoLabor comprometido → no ve datos PA1
Blast radius contenido a 1 proyecto
↓
C.7 Nginx — TLS, Rate Limiting, Webhook Perimetral
TLS
Solo TLSv1.2 + TLSv1.3
HSTS activo
Certbot no es suficiente sin config explícita
Webhook
Rate: 2r/s burst=10 sin nodelay
Content-Type: solo application/json
max_body_size: 1M
Timeouts: 10s body + header
Acceso panel
Panel N8N → solo tu IP
Nginx logs → logrotate 14 días (GDPR)
↓
C.8 Backup y Recuperación Resiliencia
Qué
pg_dump por proyecto + .env completo
N8N_ENCRYPTION_KEY incluida siempre
Configuración Nginx
Cómo
Cifrado GPG antes de salir del VPS
Destino: Hetzner Storage Box (EU, DPA)
Rotación: 7 días + 4 semanas
Verificación
Dry-run mensual obligatorio
Snapshot VPS post-hardening
RTO objetivo: < 30 minutos

C.9 Seguridad de Workflows N8N — ADR-208 Proceso
Principio
La infra contiene el daño
Los workflows SON el daño
Auditoría antes de producción
3 Preguntas de auditoría
¿Qué entra y está validado?
¿Acceso mínimo necesario?
¿Canales de salida auditados?
Vulnerabilidades PA1
HMAC placeholder → resolver pre-deploy
Capa 4 con PII → reconstruir (ADR-210)
messaggio sin sanitizar → añadir en procesador-body1
Simulación de ataque — resultado por fase (ChatGPT + Gemini + DeepSeek)
FASE 1
Reconocimiento externo
DNS lookup, escaneo de puertos, enumeración de endpoints
✓ CONTENIDO
Firewall Hetzner + Nginx → sin fuga de información. Endpoint no trivial ralentiza enumeración.
FASE 2
Abuso del webhook
Flood + replay attack + payload malicioso
⚡ PARCIAL
Rate limiting contiene el flood. Replay attack: deuda técnica — HMAC sin timestamp. Payload: pasa si HMAC es placeholder.
FASE 3-5
Compromiso N8N → PostgreSQL
Explotación de lógica de workflow → acceso a DB
⚠ ENTRA (1 proyecto)
Puede acceder a DB de UN proyecto. REVOKE CONNECT bloquea los otros. Blast radius contenido por silos.
FASE 6-8
Exfiltración + persistencia
Intento de sacar datos + establecer persistencia
✓ BLOQUEADO
DOCKER-USER bloquea salida. auditd detecta persistencia. Escape de contenedor muy difícil con no-new-privileges.
Protocolo de respuesta ante incidente — orden inapelable
PASO 1
AISLAR
Panel Hetzner → quitar acceso público al firewall. Dejar solo tu IP. Detiene exfiltración activa.
PASO 2
SNAPSHOT
Hetzner → snapshot del VPS actual. Evidencia forense antes de tocar nada. Sin esto no hay prueba documental.
PASO 3
INVESTIGAR
audit.log, docker logs, nginx logs, ps aux, netstat -tulpn, docker ps, crontab -l
PASO 4
NOTIFICAR
Si hay brecha PII: autoridad de control en 72h (GDPR Art. 33). El snapshot es la documentación del incidente.
